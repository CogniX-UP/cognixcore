Tutorial¶

Welcome to the cognixcore tutorial. Creating nodes and using them is quite simple, as you will see.

A good starting example¶

In this example, we’re going to see how to create two nodes that can work together. One node represents a Number while the other adds two Numbers.

First, our imports. All we need is the cognixcore.Node and cognixcore.PortConfig from cognixcore and the Number from numbers Python package.

1from cognixcore import (
2    Node,
3    PortConfig,
4)
5
6from numbers import Number

Now, we will define two classes that represent our nodes.

NumberNode¶

 1from cognixcore.config import NodeConfig
 2
 3class NumberNode(Node):
 4    """This node outputs any kind of number"""
 5    
 6    title='Number'
 7    version='0.1'
 8    
 9    init_outputs = [
10        PortConfig(label='num', allowed_data=Number)
11    ]
12    
13    def __init__(self, flow, config = None):
14        super().__init__(flow, config)
15        self.number = 0
16

Notice that our class extends the cognixcore.Node class. In doing so we’ve created a node blueprint. We’ve also defined two class attributes, namely cognixcore.Node.title and cognixcore.Node.version. It’s a good convention to define these attributes, as they might be used in some other context. For example, in the CogniX Editor, the title is displayed over a node’s GUI, while the version is used to filter out which nodes we want to include (no version means the node won’t be included in the editor packages)

The cognixcore.Node.init_outputs attribute is a list that contains the output ports that we want our node to have at instantiation. The cognixcore.PortConfig class is responsible for generating the port that we want. We can give the port a label and also set restrictions on its output type, by using the cognixcore.PortConfig.allowed_data attribute. The allowed_data can be None, which essentially tells cognixcore that this port can output anything, or any kind fo python type hint. Allowed data through type hinting is checked when nodes are connected as well as when outputing data through a port. These checks are made possible through the excellent beartype package.

Alls that’s left is our “processing” logic. How will the node update itself? By overriding the cognixcore.Node.update_event() method. The update_event updates the node based on an input port. In this case, the NumberNode does not have any inputs and only needs to output the internal number it represents.

AddNode¶

 1        self.set_output(0, self.number)
 2        
 3class AddNode(Node):
 4    """
 5    This is a node that adds two numbers. The numbers can 
 6    be any type of number
 7    """
 8    
 9    title = 'Add'
10    version = '0.1'
11    
12    init_inputs=[
13        PortConfig(label='num1', allowed_data=Number),
14        PortConfig(label='num2', allowed_data=Number)
15    ]
16    init_outputs=[
17        PortConfig(label='out', allowed_data=Number)
18    ]
19    
20    def __init__(self, flow, config = None):
21        super().__init__(flow, config)
22        self.result = 0
23        
24    def update_event(self, inp=-1):
25        
26        # Get the number of the current input
27        num: Number = self.input(inp)
28        if not num:
29            num = 0
30        
31        # get the number on the other port
32        other_port = 0 if inp==1 else 1
33        other_num: Number = self.input(other_port)
34        if not other_num:
35            other_num = 0
36        
37        # add them and output
38        self.result = num + other_num
39        self.set_output(0, self.result)

In similar fashion, we define an AddNode. The difference between the AddNode and the NumberNode is that this node has input ports through the cognixcore.Node.init_inputs attribute. Notice that that both of the inputs have their allowed_data set to Number. This is an essential part that will be explained later.

Lets interpret the update_event. The AddNode is updated with a given input. Using the cognixcore.Node.input() method, we’re reading the value of the connected output of a port that outputs a Number as input to the AddNode. In our case, this should be an output port of a NumberNode, since we have not defined any other nodes that output numbers. We’re then checking to see if our input is indeed a number, otherwise we set it to 0. We proceed to grab the value of the other input port in a similar fashion, add them together and output them through the only output port of the AddNode.

Using the nodes¶

Defining the nodes is a good starting point for any kind of node-based library. However, there is a need to also be able to create, connect them and eventually evaluate a graph. This would typically happen in a visual editor, such as the CogniX Editor, but can also happen in a script.

 1from cognixcore import (
 2    Flow,
 3    Session
 4)
 5
 6session = Session()
 7# Register the nodes
 8session.register_node_types([NumberNode, AddNode])
 9
10flow = session.create_flow(title='Example')
11
12# First number node
13num_one = flow.create_node(NumberNode)
14num_one.number = 25
15
16# second number node
17num_two = flow.create_node(NumberNode)
18num_two.number = 324.25
19
20# add node
21add_node = flow.create_node(AddNode)
22
23flow.connect_nodes(num_one, 0, add_node, 1, silent=False)
24flow.connect_nodes(num_two, 0, add_node, 1, silent=False)
25
26print(add_node.result)

We’re starting by creating a cognixcore.Session, then registering our nodes to the Session. cognixcore, despite offering saving and loading capabilities, does not save/load the node types. This is left to the external “package” manager that any user of the library might have to create.

After the Session is created, we create a cognixcore.Flow to the session, titled Example. We then proceed to create two NumberNodes and assign a number to each of them. Lastly, an AddNode is created and the two outputs of the number nodes are connected to the inputs of the add node. Notice that the cognixcore.Flow.connect_nodes() has a silent parameter, which is defaulted to False. When set to False, data will be propagated from the number node to the add node through the update_event and set_ouput methods, based on the kind of cognixcore.FlowExecutor executor attached to Flow. More on that later.

And that’s it, we have defined two different kind of nodes, imported them in a session, connected them and let them do their magic. Easy, isn’t it?

Now that we’ve covered the most basic aspects of the library, such as defining nodes, ports and the update_event, lets dive in the various functionalities and gotchas of cognixcore. Most of the sections below are also detailed in the concrete API documentation.

Initializations¶

This section briefly explains the initializations and de-initializations that may occur for a node.

__init__¶

As is typical with every Python class, the cognixcore.Node.__init__() constructor is called upon instantiation of a Node. This constructor is only called once.

place_event¶

The cognixcore.Node.place_event() method is invoked whenever a Node is placed inside a flow. Essentially, this defines what should happen when someone “drops” or “creates” a node inside a Flow.

Warning

This event might be deprecated in the near future in favor of the cognixcore.Node.init() method.

init¶

The cognixcore.Node.init() method is defined as an initialization point for a node, but not only at construction time. Currently, it is invoked whenever a Flow is evaluated through a cognixcore.GraphPlayer, right before the update_event. More on the GraphPlayer later.

stop¶

The cognixcore.Node.stop() method is invoked whenever a Flow has finished evaluating through a GraphPlayer.

Flow Executors - Graph Players¶

This section notes the importance of cognixcore.FlowExecutor (s) and cognixcore.GraphPlayer (s).

Per the original ryvencore, there were different algorithms to handle the cognixcore.Node.update_event() and cognixcore.Node.set_output(). The basis of these algorithms is the class cognixcore.FlowExecutor and its implementations. Each flow can have its executor set from cognixcore.Flow.set_algorithm_mode() or cognixcore.Flow.algorithm_mode(). Detailed documentation can be found in cognixcore.flow_executor and cognixcore.flow_executor.Flow.

Data Flow¶

The cognixcore.DataFlowNaive is a simple implementation of data-flow execution. Setting a node output leads to an immediate update in all successors consecutively. No runtime optimization is performed, and some types of graphs can run really slow here, especially if the include “diamonds”.

Assumptions for the graph: - no non-terminating feedback loops

Data Flow Optimized¶

Warning

Check the documentation in cognixcore.Flow. This is an original ryvencore functionality and is documented as such. However, at its current state, cognixcore is better suited for such operations through the GraphPlayer.

This is a special flow executor which implements some node functions to optimize flow execution. Whenever a new execution is invoked somewhere (some node cognixcore.Node.update or output cognixcore.Node.set_output is updated), it analyses the graph’s connected component (of successors) where the execution was invoked and creates a few data structures to reverse engineer how many input updates every node possibly receives in this execution. A node’s outputs are propagated once no input can still receive new data from a predecessor node. Therefore, while a node gets updated every time an input receives some data, every OUTPUT is only updated ONCE.

This implies that every connection is activated at most once in an execution. This can result in asymptotic speedup in large data flows compared to normal data flow execution where any two executed branches which merge again in the future result in two complete executions of anything that comes after the merge, which quickly produces exponential performance issues.

Exec Flow Naive¶

In cognixcore.ExecFlowNaive, we deviate from the common data flow paradigm. No data is propagated and instead of set_output we have cognixcore.Node.exec_output(). For more information, look at the documentation of cognixcore.Flow.

Manual Flow¶

The cognixcore.ManualFlow is an executor that doesn’t propagate any data between nodes at default. It also doesn’t handle the exec_output method. This execution algorithm simply keeps a state of the outputs that have been updated through a call to cognixcore.Node.set_output(). What we refer to as “manual execution” thus is the external evaluation of the whole flow graph, through the means of a cognixcore.GraphPlayer. While the other executors update their nodes upon connecting and disconnecting their ports, this executor is simply of a state of the current updates and the graph’s evaluation is done externally.

Graph Player¶

A cognixcore.GraphPlayer and its concrete implementation cognixcore.FlowPlayer act as evaluators of a graph as a Python program. Through this component, nodes that are formed into a flow are assembled and eventually evaluated. This class is associated with the following methods:

  • cognixcore.GraphPlayer.play(): This method is called when the graph starts playing or is evaluated and will invoked the cognixcore.Node.init() method.

  • cognixcore.GraphPlayer.pause(): This method is called when the graph is paused, but can continue. A pause state is only available on graphs that have a FrameNode incorporated, otherwise the graph will evaluate only once and exit, akin to a Python program running and finishing without any other user input.

  • cognixcore.GraphPlayer.resume(): If the graph can be paused, then it can also be resumed. This method resumes the execution of the current graph.

  • cognixcore.GraphPlayer.stop(): This method stops the graph completely and is called on demand if the graph contains FrameNode (s) or automatically if it doesn’t. Invokes the cognixcore.Node.stop() method.

How does the default cognixcore.FlowPlayer work? The first step is to find all the active nodes in a flow. A node is considered active when it has connections or is a FrameNode Subsequently, these nodes are separated into two categories. The root nodes and the frame nodes. The root nodes are nodes that have no input ports connected to them. Essentially, they act as a starting point for the evaluation. The frame nodes will be evaluated per frame.

The active nodes are then initialized. Each node may have resources that need to be initialized based on their configuration. The nodes are then evaluated based on a Breadth First Search (BFS) algorithm to ensure that each node is evaluated only once for each connected input. After the evaluation, the algorithm checks the number of active frame nodes. If there are frame nodes present in the graph that need evaluation, they are evaluated. The successors of the frame nodes are then updated as if they were the root nodes. This frame node evaluation process is repeated until the user manually stops the graph or every frame node has finished its operation based on an internal condition.

A simple example on how to play a graph is provided below. Note that the executor is automatically changed to ManualFlow, but will be changed back to the previous algorithm node when the graph evaluation is stopped.

 1from cognixcore import (
 2    Session,
 3    FlowAlg,
 4)
 5
 6session = Session()
 7flow = session.create_flow("Example Flow")
 8flow.algorithm_mode = FlowAlg.DATA_OPT
 9
10# This will manually set it to a ManualFlow executor
11session.play_flow("Example Flow")

It’s always better to use the Session to evaluate a graph through a GraphPlayer rather than through the GraphPlayer or the Flow, as these methods also invoke certain events and handle threading better, if the player is invoked in another thread (check the on_other_thread) parameter of the cognixcore.Session.play_flow() method.

Frame Node¶

The cognixcore.FrameNode is a special type of node that in conjuction with a GraphPlayer. A frame node is evaluated once per “frame”. A frame in a GraphPlayer denotes an execution step where the player needs to update all its frame nodes, through the approach described previously. Lets see a basic example.

 1from cognixcore import (
 2    Session,
 3    Node,
 4    FrameNode,
 5    PortConfig
 6)
 7from random import random
 8from numbers import Number
 9
10class NumberGeneratorNode(FrameNode):
11    """Generates a number"""
12    
13    title='Number Generator'
14    version='v0.1'
15    
16    init_outputs = [
17        PortConfig(label='num', allowed_data=Number)
18    ]
19    
20    def __init__(self, flow):
21        super().__init__(flow)
22        self.scale = 2
23    
24    def frame_update(self):
25        num = self.scale * random()
26        self.set_output(0, num)
27    
28session = Session()
29session.register_node_types([NumberGeneratorNode, AddNode])
30
31flow = session.create_flow('Random Number Generation')
32rnd_node = flow.create_node(NumberGeneratorNode)
33add_node = flow.create_node(AddNode)
34
35flow.connect_nodes(rnd_node, 0, add_node, 0)
36
37session.play_flow(
38    'Random Number Generation',
39    on_other_thread=True,
40)

We begin by defining a FrameNode. This node has only one output. We then implement its cognixcore.FrameNode.frame_update() method. This method will be called one per frame, generating a random number and outputting it through its port. We then create a session, register our node and instantiate it, along with the AddNode from our first example. We then connect the nodes and play the flow. If we want the current flow of execution to not be stopped, we should pass True to the on_other_thread parameter, as shown.

Quality of Life¶

The above sections include the main functionalities of the cognixcore library. In this section, we’ll focus on some QoL utilities that work out of the box with the CogniX Editor.

Loading/saving¶

The nodes support state saving and loading through the cognixcore.Node.load() and cognixcore.Node.data() methods. Additionally, there are similar methods that can be found for the Session and Flow, however these should rarely be overriden.

Progress State¶

The cognixcore.ProgressState is a class that helps with setting the state of a node. In the context of simply scripting, this doesn’t do much. However, it exists as an out of the box solution for any kind of GUI library that needs to define ongoing progress for a node. For example, the CogniX Editor automatically utilizes this to set a GUI that reacts to progress changes.

Loggers¶

Each node and flow have a logger through the cognixcore.addons.logging module, based on Python’s logging module. This can be accessed through their respective properties (cognixcore.Node.logger and cognixcore.Flow.logger).

Variables¶

Each node has access to a cognixcore.addons.variables.VarsAddon for reading and setting the state of dynamic variables. Refer to the documentation for further details.

Node Actions¶

A node can define actions through the cognixcore.NodeAction and the cognixcore.Node.add_action() and cognixcore.Node.add_generic_action() methods. In doing so, a context menu is populated for a node’s GUI with the actions defined in the CogniX Editor. You would typically define the actions in the __init__ method of a node.

Dynamic ports

The static cognixcore.Node.init_inputs and cognixcore.Node.init_outputs initialization lists only work when a node is instantiated but not loaded through data. A node instance supports adding or removing ports dynamically at runtime. This is done through the following methods:

  • cognixcore.Node.create_input()

  • cognixcore.Node.delete_output()

  • cognixcore.Node.create_output()

  • cognixcore.Node.delete_output()

Node Configuration¶

A node can have a configuration attached to it. This is done by either defining an internal to the node class explicitly named Config or by setting the cognixcore.Node.inner_config_type type to a specific node config. The base class for a configuration is the cognixcore.NodeConfig class. A developer can override this class to create a custom configurable class.

Configuration by Traits¶

The standard implementation for a node configuration is the cognixcore.config.traits.NodeTraitsConfig class. It utilizes the Traits and Traits UI for out of the box data validation and GUI generation (for the CogniX Editor). This implementation allows for dynamic nesting of multiple configurations, all of which are saveable/loadable. Traits with the “CX” prefix should be preferred over typical traits (such as CX_Float instead of Float), because they invoke their changed event when pressing the enter button. This is mainly a CogniX Editor concern, coupled with its undo functionality. This section only serves as an introduction to traits and its cognixcore configuration counterpart. Please refer to the original traits documentation for additional user-manuals and tutorials.

Port list¶

Along with the standard implementation for configuration, the cognixcore.config.traits module offers a class that makes it easier to incorporate dynamic port handling. This class is cognixcore.config.traits.PortList class.

RESTful API¶

cognixcore also offers the ability to run a RESTful API for a session. This can be done by accessing the API through the cognixcore.Session.rest_api property. For more info regarding the API, you can run a session and the api and check the docs through localhost:<port_number>/docs.

A better example¶

To incorporate the above, we will iterate upon our first example to optimize the code and make it work better with the CogniX Editor.

First, our imports. We’re using quite a lot more than before.

 2from typing import Self
 3
 4from cognixcore import (
 5    Node,
 6    PortConfig,
 7    ProgressState,
 8)
 9from cognixcore.config.traits import *

11from numbers import Number

Number Node¶

 1class NumberNode(Node):
 2    """This node outputs any kind of number"""
 3    
 4    title='Number'
 5    version='0.1'
 6    
 7    class Config(NodeTraitsConfig):
 8        mode: str = Enum('random', 'int', 'float', desc="mode for number generation")
 9        int_num = CX_Int(0, visible_when='mode=="int"', desc="int number")
10        float_num = CX_Int(0.0, visible_when='mode=="float"', desc="float number")
11        
12        def number(self) -> Number:
13            if self.mode == 'random':
14                return random()
15            elif self.mode == 'int':
16                return self.int_num
17            else:
18                return self.float_num
19    
20    init_outputs = [
21        PortConfig('num', allowed_data=Number)
22    ]
23    
24    @property
25    def config(self) -> Config:
26        return self._config
27        
28    def update_event(self, inp=-1):
29        self.progress = ProgressState(1, -1, "Outputing number")
30        self.set_output(0, self.config.number())
31        self.progress = None

Notice that the NumberNode now contains a configuration class. In this, we define a mode, giving our node different types of number generation. There is a random mode, where the node returns a random number. Then there are two other modes, int and float for integers and floats respectively. This can also be seen from the int_num and float_num attributes.

In the update_event, we now also set a progress to indicate that our node has started generating the number and that it has finished. Granted, this operation will be really fast and hence won’t be visible in the CogniX Editor or any other GUI implementation for a progress state.

Add Node¶

 1class AddNode(Node):
 2    """
 3    This is a node that adds two numbers. The numbers can 
 4    be any type of number
 5    """
 6    
 7    title = 'Add'
 8    version = '0.1'
 9    
10    class Config(NodeTraitsConfig):
11        ports: PortList = Instance(
12            PortList,
13            lambda: PortList(
14                    list_type = PortList.ListType.INPUTS,
15                    inp_params = PortList.Params(
16                        allowed_data=Number    
17                    ),
18                )
19        )
20        
21        def set_ports(self, ports: list[str]):
22            self.port.ports = ports
23        
24    init_outputs=[
25        PortConfig(label='out', allowed_data=Number)
26    ]
27    
28    def __init__(self, flow, config = None):
29        super().__init__(flow, config)
30        self.result = 0
31    
32    def init(self):
33        self.first_res = True
34    
35    # it's better to redefine the config for intellisense purposes
36    @property
37    def config(self) -> Config:
38        return self._config
39    
40    def update_event(self, inp=-1):
41        
42        prev_result = self.result
43        
44        self.result = 0
45        for i in range(self.num_inputs):
46            val = self.input(i)
47            if not val:
48                val = 0
49            self.result += val

The AddNode now also contains a configuration class, with only one configuration. This configuration allows for easily and dynamically changing the ports of a node. Notice that for this node, we’re only altering the inputs. Thus, we have removed the init_inputs static attribute, since it is no longer needed.

Our update_event logic is now written for the AddNode to work with an arbitrary number of inputs. We’re also making sure to output a result only if the result is different than a previous iteration. These implementation details might not work for you and you might choose to output the result despite it being the same as a previous iteration. This is specific to the role of the node and the developer behind it.

Finally, note that we’re also logging the result. This would show in the CogniX Editor, or any other GUI that taps into the logging system of cognixcore.

Using the nodes¶

 1from cognixcore import (
 2    Flow,
 3    Session
 4)
 5
 6session = Session()
 7# Register the nodes
 8session.register_node_types([NumberNode, AddNode])
 9
10flow = session.create_flow(title='Example')
11
12# First number node
13num_one = flow.create_node(NumberNode)
14num_one.config.mode = 'random'
15
16# second number node
17num_two = flow.create_node(NumberNode)
18num_two.config.mode = 'int'
19num_two.config.int_num = 25
20
21# add node
22add_node: AddNode = flow.create_node(AddNode)
23# dynamically change the ports through the port list
24add_node.config.set_ports(['port_one', 'port_two'])
25
26flow.connect_nodes(num_one, 0, add_node, 1, silent=False)
27flow.connect_nodes(num_two, 0, add_node, 1, silent=False)
28
29print(add_node.result)

Finally, we’re using the nodes in a scripting rather than visual manner. We’re creating a session, registering the nodes and creating a flow. We’re instantiating two NumberNode (s) and one AddNode. The first number node generates a random number while the second outputs the number 25. We’re then making sure that the add node has two ports, which we set dynamically. Finally, we connect the nodes that work through the default DataFlowNaive algorithm.

Better yet, instead of relying on the DataFlowNaive algorithm, we can evaluate our graph as a Python program, like this:
